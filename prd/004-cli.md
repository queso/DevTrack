# PRD 004: CLI Tool

## Summary

The DevTrack CLI. Generated from the OpenAPI spec via swagger-jack, with a thin layer of repo-aware convenience commands on top. This is how hooks talk to the API, how you register projects, and how you query state from the terminal.

## Problem

The dashboard is great for the overview, but when you're in a terminal working on a project, you don't want to switch to a browser to check status or record an event. The CLI meets you where you are. It's also the transport layer that hooks use to push state to the API.

## Architecture

Two layers:

1. **Auto-generated API client** — swagger-jack reads the OpenAPI spec from PRD 001 and generates typed commands for every endpoint. This is the bulk of the CLI and stays in sync with the API automatically.

2. **Convenience commands** — a thin wrapper that reads `project.yaml`, infers context from the current repo (git branch, remote URL), and calls the generated commands with the right arguments. These are the human-friendly commands.

## Convenience Commands

These commands are repo-aware. Run them from inside a tracked repo and they auto-detect context from `project.yaml` and git state.

### `devtrack register`

Register the current repo with DevTrack.

- Reads `project.yaml` from repo root
- Auto-detects `repo_url` from `git remote get-url origin`
- POSTs to `/api/v1/projects`
- Validates manifest before sending
- Updates existing project if already registered
- Offers to install hooks

### `devtrack status`

Show current project status. Quick terminal check without leaving your editor.

```
$ devtrack status

  picking-app (sdlc · arcanelayer)
  ────────────────────────────────
  PRD:    Barcode Scanning Support (3/5 items done)
  Branch: feat/barcode-scanning
  PR:     #42 — open, checks passing
  Last:   2 hours ago
```

### `devtrack dashboard`

Cross-project overview from the terminal. The CLI equivalent of the home page.

```
$ devtrack dashboard

  12 projects · 3 need attention · 2 PRs open

  ● picking-app          Adding barcode scanning — 3/5 done     2h ago
  ● aiteam-brand         Landing page redesign — 5/5 done, PR ready   4h ago
  ○ k8s-infra            Migrate to Cilium — 1/3 done           2d ago
  ◌ arcanelayer-store    No active PRD                          14d ago
```

Legend: ● active today, ○ this week, ◌ stale

### `devtrack prs`

Show all open PRs across projects.

```
$ devtrack prs

  PR QUEUE (3 open)

  picking-app     #42  Add barcode scanning     open    2h
  aiteam-brand    #15  Landing page redesign     approved  4h
  k8s-infra       #8   Cilium network policies   draft   1d
```

### `devtrack event <type>`

Record an event. Primarily used by hooks, but available manually.

```bash
devtrack event commit --branch main --hash abc123 --message "Fix picking bug"
devtrack event deploy --environment production
devtrack event prd-updated --prd-id <id>
```

### `devtrack sync`

Force a full sync of the current repo's state to the API:
- Re-reads `project.yaml` and updates project
- Scans `prd_path` and syncs all PRDs + work items
- Scans `content_path`/`draft_path` and syncs content items
- Fetches open PRs from GitHub and syncs

Useful for initial setup or if hooks missed something.

### `devtrack ideas`

Content workflow shortcut. Manage the idea backlog.

```bash
devtrack ideas                          # list ideas for current project
devtrack ideas --all                    # list ideas across all content projects
devtrack ideas add "Why PRDs matter"    # add an idea
devtrack ideas add "Solo dev workflow" --tags blog,ai --summary "How I manage 16 repos"
devtrack ideas promote <id>             # promote idea to draft (scaffolds file in repo)
```

### `devtrack hooks install`

Install Claude Code hooks for the current repo.
- Reads `project.yaml` to determine workflow type
- Writes hook scripts to `.claude/hooks/`
- Configures `.claude/settings.json` for hook events

### `devtrack hooks test`

Verify hooks are installed and can reach the API.

## Configuration

Global config at `~/.devtrack/config.yaml`:

```yaml
api_url: https://devtrack.owenslabs.com/api/v1
api_key: dt_xxxxxxxxxxxx
```

Environment variables override config file:
- `DEVTRACK_API_URL`
- `DEVTRACK_API_KEY`

## Generated Commands

swagger-jack generates commands for every API endpoint. These are the "raw" commands:

```bash
devtrack api projects list
devtrack api projects get <id>
devtrack api projects create --name "foo" --workflow sdlc ...
devtrack api prds list --project-id <id>
devtrack api events list --project-id <id> --type commit
devtrack api prs list --status open
# etc.
```

These mirror the API 1:1. The convenience commands above are sugar on top of these.

## Output Formatting

- Default: human-readable tables/summaries (like the examples above)
- `--json` flag on any command: raw JSON output for scripting
- `--quiet` flag: minimal output (just IDs or status codes)
- Colors and formatting via terminal capabilities, gracefully degrade in non-TTY

## Error Handling

- API unreachable: clear error message with suggestion to check config
- Invalid manifest: show validation errors with line numbers
- Auth failure: prompt to check API key
- All errors to stderr, never mixed with data output
